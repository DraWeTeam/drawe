# 0008 — 운영 비용: 측정 후 "최적화 안 함"

상태: 채택 (2026-07-02)

## 맥락

제품 운영 비용(작가가 /guide 호출할 때마다 도는 vertex VLM + LLM)을 줄일 여지를 검토했다. 출발점은
**"VLM 비싸다"는 일반론**이었고 실제 비용은 미측정이었다. [0002]의 "추측 최적화 금지·측정 먼저" 원칙에
따라, 손대기 전에 **/guide 한 번에 뭐가 실제로 도는지 코드 경로로 정찰**했다(읽기 전용).

## 결정

- **정찰 결과 = 큰 절약이 이미 구현돼 있음**, 그래서 **개입하지 않는다**(근거 있는 비개입):
  - **VLM 삼중 게이트** — env 플래그 + creds + (CLIP 확신 밴드 / pose-tier). 확신 figure·풍경·빈 캔버스엔
    **0콜**, 애매 이미지만 2~4콜. 매 요청 전부 부르지 않는다.
  - **2단 캐시** — L1(in-proc 512) + L2(Redis, 30일 TTL), 키=이미지 해시+모델ID. 같은 그림 재업로드 0콜,
    요청 내 여러 축이 같은 신호 공유(dedup).
  - **공짜 계산** — scorer 12개(numpy/opencv)·SVG 카드(`.format()`)·DB(이벤트 로그)는 API 비용 0.
  - **LLM = coach 1콜**(단일 큰 프롬프트로 전부 생성). 개인화(user_id·message·growth)라 **캐시 불가**.
- **"안 한 것도 결정"** — 이미 게이팅·캐싱이 최적에 가까워, 측정 없이 추가하면 중복·회귀 위험만 있다.
- **남은 레버는 배포 후 실측으로** — 유일한 상시 비용인 coach LLM의 모델 티어 조정은 품질 트레이드오프라
  프로덕션 실측 데이터가 있어야 판단한다(지금 추측으로 낮추지 않는다).

## 기각한 대안

- **"VLM 비싸다"로 즉시 게이팅/캐싱을 더 넣는다.** 기각: 이미 구현돼 있다 — 측정 없이 손대면 중복 코드·회귀.
  일반론이 아니라 코드가 근거다("배선 ≠ 판정"의 비용판).
- **coach LLM 결과 캐싱.** 기각: user별 성장 이력·메시지에 개인화돼 있어 캐시하면 연속성이 깨진다.
- **지금 모델 티어를 낮춘다.** 보류(기각 아님): 최대 레버지만 품질 손실은 실측 없이 못 판단 → 배포 후.

## 결과

- (+) 없는 낭비를 만들지 않는다 — 측정이 "이미 최적에 가까움"을 확인. 엔지니어링 시간을 안 쓰는 게 옳은 선택.
- (−) 남은 레버(모델 티어)는 미착수 상태 — 배포 후 실측 대기.
- **★배포 prep 확인 항목(비용보다 인프라):** ① **prod에 Redis L2가 보장돼야** 한다 — 없으면 캐시가 L1(512·
  재시작 소실)로 격하돼 같은 그림 재계산이 급증(실질 최대 비용 방어선). ② **prod VLM 플래그 드리프트** —
  configmap에 플래그 누락 시 VLM이 아예 안 떠 비용 ~0이 되지만 정확도 기능도 dormant가 된다(비용/기능 트레이드오프).
