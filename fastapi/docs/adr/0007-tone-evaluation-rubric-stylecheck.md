# 0007 — 톤 평가: Rubric(사람) + Style Check(자동) 분리

상태: 채택 (2026-07-02)

## 맥락

정확도는 자동 eval + 블라인드 골든 라벨로 측정했다([0002]). 그러나 **톤**(가이드 발화가 러닝메이트/
작업실 선배답게 읽히는가)은 미측정이었다 — "VLM 비싸다" 같은 일반론처럼 톤도 "친근하게 잘 나온다"는
인상만 있었다. 톤은 성격이 이중적이다: **주관**(존중·따뜻함·러닝메이트다움)과 **객관**(문장 길이·전문용어
수·한 번에 던지는 개선점 수)이 섞여 있다. 정확도에서 쓴 "자동 지표 + 사람 판단" 이분법을 톤에 적용한다.

## 결정

- **두 층으로 분리해 섞지 않는다.**
  - **Style Check(자동·객관)** — 발화별 기계 추출: 평균 문장 길이, 개선점 개수(몇 축 던지나), 전문용어
    목록+[풀림/안됨], 영어/raw-id 개수, 행동 제안 유무. 재현적이라 사람이 셀 필요 없다.
  - **Rubric(사람 판단·6기준)** — ①존댓말 ②인정/공감 먼저(★가짜 칭찬 아닌가) ③어려운 용어 풀이
    ④하나의 핵심 개선점 중심 ⑤행동 다음 단계 ⑥판단 근거 한 문장. **AI 점수는 참고용, 최종 판단은 사람.**
- **자동 점수만으로 결론 금지** — 톤 목표(러닝메이트다움)는 주관이라 자동 점수 단독은 골든fit과 같은 함정.
- **반복 평가 자산화** — 수집·지표 도구로 프롬프트 수정 전후를 같은 지표로 비교(회귀 방지).

## 기각한 대안

- **AI 점수만으로 톤 판정.** 기각: 주관 항목(가짜 칭찬 여부·따뜻함)은 자동화가 신뢰 못 한다. 자동 점수에
  프롬프트를 맞추면 "점수는 오르는데 실제 톤은 나빠지는" overfit([0002]와 같은 위험).
- **Rubric만(객관 지표 없이).** 기각: 문장 길이·개선점 수·raw-id 잔존 같은 객관 지표는 자동이 정확·재현적
  이다(실제로 Style Check가 raw-id 누출을 자동 검출→[cdb575e]). 사람이 셀 이유가 없다.
- **정확도 골든셋에 톤도 얹기.** 기각: 정확도는 정답(축)이 있지만 톤은 정답이 없다 — 다른 평가다.

## 결과

- (+) 톤도 정확도처럼 프롬프트 수정 전후 반복 평가 가능. 주관/객관을 분리해 각 방식의 강점만 쓴다.
- (+) 첫 실측(15발화)이 구체 이슈를 잡음: 진단 모드가 top-3를 완전 전개(Rubric④ 위반 소지) · landscape
  용어(원경/근경/채도) 미풀이(Rubric③) · raw-id 누출(Style Check가 자동 포착).
- (−) Rubric은 사람 시간이 든다(6기준 × N발화). AI 점수는 참고선까지만.
- **한계(정직히):** 수집·지표 스크립트(`collect.py`·`analyze.py`)는 아직 세션 scratchpad에만 있고 **repo에
  커밋되지 않았다**. "반복 평가 자산"이 되려면 후속 커밋이 필요하다 — 현재는 방법론 결정만 확립된 상태.
