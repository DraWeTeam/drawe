# 0006 — 골든셋 자산 관리: public 레포엔 텍스트만, 이미지는 제외

상태: 채택 (2026-07-01)

## 맥락

골든셋은 정확도 평가의 기준이다 — 이미지(89장) + 블라인드 라벨·기대값([0002]) + 무결성 해시.
그런데 이 저장소는 **public 모노레포**이고 이미지는 **Imagen으로 생성**했다: 약관·라이선스 노출 우려 +
바이너리가 히스토리를 비대하게 만든다(수십 MB). 동시에 **평가 재현성**을 위해 라벨·기대값·해시는
반드시 버전관리돼야 한다(라벨이 없으면 스코어를 재현할 수 없다).

## 결정

- **텍스트 자산만 커밋** — `labels.json`(블라인드 라벨·기대값), `expansion_meta.json`, `golden_images.sha256`,
  생성 스크립트를 tracked로. AI 생성 `.png`는 커밋 안 함([341b16d]).
- **이미지는 해시로 대리 추적** — `golden_images.sha256`(tracked)이 이미지 기준 바이트를 고정한다. 이미지
  자체는 없어도 다른 머신에 복원 후 바이트 일치를 검증할 수 있다.
- **.gitignore 이중 차단** — `golden-set-kit/images/` + `golden-set-kit/**/*.{png,jpg,jpeg}`. 실수로도
  AI 이미지가 public 히스토리에 남지 않게.
- **이미지는 로컬 보존(무료 백업)** — S3 미사용. 유실 대비는 로컬 별도 복사/팀 드라이브로.

## 기각한 대안

- **이미지도 커밋.** 기각: public 노출(Imagen 약관 회피) + 모노레포 히스토리 영구 비대. 되돌리려면 history rewrite.
- **private 레포/서브모듈로 분리.** 기각: 팀 접근·CI 복잡도 증가, 별도 트랙 관리 부담. 텍스트 자산은
  본 코드와 함께 있어야 재현이 자연스럽다.
- **S3/클라우드 스토리지.** 기각: 비용·인프라. 89장 규모엔 과하다 — 필요해지면 그때.
- **git-LFS.** 보류(기각 아님): 규모가 커지면 이전 대상으로 README에 명시. 지금은 sha256 대리 추적으로 충분.

## 결과

- (+) 라이선스 깨끗 + public 안전 + 평가 재현성(라벨·해시 tracked).
- (+) 실수 방지 이중 gitignore로 AI 이미지 누출 경로 차단.
- (−) 이미지가 **로컬 단일 사본**이라 유실 위험 — sha256로 무결성은 검증 가능하나 원본 복구는 재생성 필요.
  이 비용을 인지하고 무료 백업을 권장한다.
