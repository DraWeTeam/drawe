# 0002 — 정확도 작업 방법론: single-change isolation + 평가 위생

상태: 채택 (2026-06-30)

## 맥락

가이드의 축 정확도는 신호·임계·라우팅이 얽혀 있다. 변경을 한꺼번에 하면 *어느 게 효과인지* 못 가리고,
측정 없는 fix는 추측이다(실제로 추측이 여러 번 틀렸다 — `prominence≈0.5` 고정도 "상수다"를 코드로
확인하고서야 버그로 드러났다). 평가가 허술하면 "좋아진 척"하기 쉽다: 정확도와 안정성을 혼동하거나,
테스트셋(골든 15장)에 임계를 맞춰버린다.

## 결정

- **single-change delta isolation** — 한 번에 한 가지만 바꾼다. 그래야 delta가 그 변경에 귀속된다.
- **측정 먼저** — fix 전 코드/실측으로 근거를 확인하고, **움직일 이미지를 예측 → 적용 → 실제 일치 확인**.
  예상 밖이 움직이면 그건 커플링 신호다.
- **평가 위생** — ① 블라인드 라벨을 출력 보기 *전*에 붙인다. ② 정확도 ≠ 안정성을 분리한다(명확/애매
  태깅 + N회 반복·캐시 flush로 안정성을 따로 검증 — 캐시 안 끄면 가짜 완벽안정). ③ 임계는 **분포의
  자연 갭/분위수**에서 잡고 **골든 점수에 fitting 금지**.
- **측정 도구 버그는 측정 대상보다 먼저** 고친다(track-aware 채점판의 비대칭을 그 위에서 평가할 축보다 먼저 수정).

## 기각한 대안

- **여러 축을 한 세션에 묶기(속도).** 기각: delta 귀속 불가 — 무엇이 효과였는지 영영 모른다. 수동 루프가
  느린 건 사실이나, 그건 측정 자동화·섀도우로 풀 문제지 measurability를 버려서 풀 게 아니다.
- **임계(`_HIGH` 등)를 올려 골든 한 장 살리기.** 기각: overfit — 문턱을 억지로 올리면 다른 케이스(figure_004)까지
  끌려와 깨진다. 임계는 한 장 살리기가 아니라 분포에서.
- **출력을 보고 라벨을 정하기.** 기각: 확증 편향. 블라인드가 먼저다.

## 결과

- (+) 매 변경 효과가 측정으로 귀속되고, 회귀가 조기 발견된다(`track_aware_eval`).
- (+) "좋아진 척"(안정성과 혼동된 정확도, overfit)이 차단된다.
- (−) 한 세션에 한두 축밖에 못 나간다(느림) → 측정 인프라(자동 회귀·섀도우)로 완화.
- **한계(중요):** 골든 15장은 통계 표본으로 작고, **테스트셋이 곧 개발셋**이라 일반화에 태생적으로 낙관적이다.
  그래서 이 규율(특히 골든fit 금지)이 일반화의 사실상 유일한 방어선이다 — 규율을 깨면 숫자는 올라도 실효는 안 따라온다.

## 프로덕션 검증 (갱신 2026-07-02): "배선 ≠ 판정"이 실제로 잡은 것

이 방법론의 핵심 전제 — **코드에 신호·지시가 있어도 실제로 작동하는지는 별개**(특히 LLM은 비결정적) —
는 UI 진입 전 실측에서 반복 적중했다. 격리 단위검증만으로는 통과하지만 full 경로에서 깨지는 사례들:

| 사례 | 격리 검증 | full 경로 실측 | 조치(커밋) |
|---|---|---|---|
| color AND-gate | 신호 계산됨 | 실발화 **0/69**(휴면 — 임계가 분포 밖) | 분포 갭에서 재튜닝 `e12e35d` |
| 벽축 pseudo-axis 카드 | `floor_svg`·`/guide-asset` 렌더 OK | **HTTP 500** (`routes.py` `personas[0]` IndexError) | 빈-리스트 가드 `b4b5787` |
| raw axis-id 라벨 | `LABELS.get` 경로 수정 완료 | **여전히 누출**(LLM이 next_steps_note에 raw id echo) | 표시 문자열 정규화 `cdb575e` |
| 그 정규화의 `\b` 정규식 | — | **내 수정이 한글 조사("id을")에서 실패** — unit test가 자경 | lookaround 교정 `cdb575e` |
| 연속발화(prompts.py:87-91 recurred) | LLM 프롬프트에 지시 있음 | per-block "지난번에도" **0/5 발화**(growth 채널이 실제 커버) | 프롬프트≠발화 확인, 표면화는 UI로 |

교훈: **"배선 성공(신호가 계산됨·코드가 있음) ≠ 판정 성공(쓸 만한 발화가 실제로 나감)"**. 특히 LLM 경유
경로는 코드 수정 후에도 격리검증을 우회하므로 full 경로 실측이 필수다. 이 표의 5건은 전부 단위검증을
통과한 뒤 실측에서 드러났다 — 방법론이 없었다면 프로덕션에 그대로 나갔을 것들이다.
